# Agent Core Rules

## 1. 代码质量红线 (Code Quality Bottom-Line)
- **拒绝“屎山”代码**：任何情况下都不允许随意堆砌代码，必须保持代码结构清晰、逻辑严谨、可读性高。
- **避免重复造轮子 (DRY)**：不要复制粘贴代码（Don't Repeat Yourself）。遇到重复逻辑，必须抽取为公共的方法或类。
- **命名即注释**：变量名、方法名必须高度自解释（Self-descriptive）。通过清晰的英文命名表达用途，不要用 a, b, temp 等无意义词汇。

## 2. 纯粹的 Bug 修复与容错 (Bug Fixing & Error Handling)
- **直击问题根因**：修 bug 时必须找出**真正导致出错的地方**并从源头解决！
- **拒绝无脑兜底**：禁止仅仅为了掩盖报错而到处加 `if (xxx == null) return;` 或无意义的 `try-catch`。不要为错误兜底，要消灭错误发生的原因。
- **快速失败 (Fail-Fast)**：当遇到不可挽回且不应该存在的非法状态时，抛出明确的异常（如 `InvalidOperationException` 或加上日志），而不是让程序带着错误状态默默运行到下个阶段。

## 3. 架构与设计准则 (Architecture & Design Principles)
- **简洁优雅**：代码应当如诗般优美，去除一切不必要的复杂度和冗余操作，用最简单的逻辑实现最强大的功能。禁止多层 Fallback（如一层套一层的容错）。
- **模块化设计**：坚持单一职责原则（SRP），将功能拆分为独立模块，使其易于复用和维护。
- **低耦合、高内聚**：严格控制模块/类之间的依赖关系，通过优秀的接口设计确保系统的灵活性。
- **最小可见性原则**：所有的变量、方法或类，默认使用能用的最小访问权限（如 `private`, `internal`, `readonly`）。绝不随意暴露内部状态。

## 4. 性能与资源意识 (Performance & Resource Awareness)
- **热路径克制分配与阻塞**：网关转发、调度、流式响应、计费记录和后台队列属于高频路径，避免无意义分配、重复 JSON 解析、重复数据库查询和无界 goroutine。
- **缓存与生命周期清晰**：缓存、并发槽位、Redis/DB 状态、worker pool 和后台任务必须有明确失效、释放和停止路径，避免资源泄漏或陈旧状态继续参与调度。

## 5. 行为与协作规范 (Behavioral Norms)
- **三思而后行**：在敲代码之前先构思好优秀的架构，确保方案是对的再进行开发。
- **有意义的注释**：不要用注释重复代码在做什么（What），而是解释为什么要这么做（Why），尤其是兼容旧数据、上游协议差异、迁移期行为和安全边界。
- **遵守现有风格**：融入当前项目的代码风格，保持全局一致性。
- **文档同步是交付的一部分**：任何涉及接口、配置、脚本、架构、部署、开发流程、分支工作流或前端规范的改动，工作前必须先阅读 `/home/senran/Desktop/sub2api/docs/README.md`，再按其文档地图检查 `/home/senran/Desktop/sub2api/docs` 中对应文档；如果没有现成文档，就补充新文档并在 `docs/README.md` 登记入口。
